P4Runtime is a gRPC-based control plane API defined by the P4 Language Consortium. Unlike the Thrift API, P4Runtime is **program-independent**: it works with any P4 program by using a P4Info file generated by the compiler to describe the tables, actions, counters, and other objects in your program.

BMv2 supports P4Runtime through the `simple_switch_grpc` target. If you are using the standard `simple_switch` target, see [Runtime CLI](/control-plane/runtime-cli) instead.

## P4Runtime vs Thrift API

| Capability | Thrift (simple\_switch) | P4Runtime (simple\_switch\_grpc) |
|---|---|---|
| Protocol | Apache Thrift RPC | gRPC (HTTP/2 + Protocol Buffers) |
| Program-independent API | No | Yes |
| Load P4 program at runtime | No | Yes — `SetForwardingPipelineConfig` |
| Packet-in / packet-out | No | Yes — via `StreamChannel` |
| Idle timeout notifications | No | Yes |
| Standardized across targets | No | Yes |
| Default port | 9090 | 9559 |

<Note>
  P4Runtime and the Thrift API maintain independent state. Writing a table entry via one API makes it invisible to the other. If you enable the Thrift server on `simple_switch_grpc`, use it only for read-only inspection.
</Note>

## Running simple_switch_grpc

Start `simple_switch_grpc` with `--no-p4` to defer pipeline loading to the `SetForwardingPipelineConfig` RPC. Pass target-specific flags after `--`.

```bash
sudo simple_switch_grpc \
    --no-p4 \
    -i 0@veth0 \
    -i 1@veth1 \
    -- \
    --grpc-server-addr 0.0.0.0:9559 \
    --cpu-port 64
```

### Key flags

| Flag | Default | Description |
|---|---|---|
| `--grpc-server-addr <addr>` | `0.0.0.0:9559` | Address and port for the P4Runtime gRPC server. Accepts `host:port`, `[::1]:port`, etc. |
| `--cpu-port <port>` | _(none)_ | CPU port for packet-in / packet-out. Required to use `StreamChannel` for packet I/O. Must be 1–510 when using v1model. |
| `--no-p4` | — | Start without a P4 program. All packets are dropped until a pipeline is loaded. |

<Warning>
  Do not use CPU port `0` or `511`. Port `511` is the default drop port in v1model. If you need port `511` as your CPU port, override the drop port with `--drop-port`.
</Warning>

## Loading a P4 program at runtime

When `simple_switch_grpc` starts with `--no-p4`, it drops all packets until you call the `SetForwardingPipelineConfig` RPC. This RPC loads the P4 pipeline without restarting the process.

<Steps>
  <Step title="Compile your P4 program">
    Use `p4c` to produce both the BMv2 JSON file and the P4Info text proto:

    ```bash
    p4c --target bmv2 --arch v1model --std p4-16 \
        my_program.p4 \
        -o my_program.json \
        --p4runtime-files my_program.p4info.txtpb
    ```
  </Step>
  <Step title="Call SetForwardingPipelineConfig">
    From your controller, issue a `SetForwardingPipelineConfig` RPC with action `VERIFY_AND_COMMIT`. Supply the P4Info proto and the JSON file contents as the `p4_device_config` field.

    Using [p4runtime-shell](https://github.com/p4lang/p4runtime-shell):

    ```bash
    p4runtime-shell \
        --grpc-addr 0.0.0.0:9559 \
        --device-id 0 \
        --election-id 0,1 \
        --config my_program.p4info.txtpb,my_program.json
    ```

    Using a Python P4Runtime client:

    ```python
    import p4runtime_lib.simple_controller as sc

    sc.program_device(
        p4info_file_path="my_program.p4info.txtpb",
        bmv2_json_file_path="my_program.json",
        sw_addr="127.0.0.1:9559",
        device_id=0,
    )
    ```
  </Step>
  <Step title="Install table entries">
    Once the pipeline is loaded, use `WriteRequest` RPCs to install table entries, configure multicast groups, and manage other data plane state.
  </Step>
</Steps>

## Packet-in and packet-out

When you start `simple_switch_grpc` with `--cpu-port`, the P4Runtime `StreamChannel` bidirectional stream carries CPU packet I/O between the controller and the switch.

- **Packet-in**: when the P4 program sends a packet to the CPU port (e.g. via `egress_spec = CPU_PORT`), `simple_switch_grpc` wraps it in a `PacketIn` message and sends it to the controller over `StreamChannel`.
- **Packet-out**: the controller sends a `PacketOut` message over `StreamChannel`. The switch injects it into the pipeline as if it arrived on the CPU port.

```python
# Receive packet-in
for msg in stream_client:
    if msg.HasField("packet"):
        pkt = msg.packet
        print(f"Received packet-in: {pkt.payload.hex()}")

# Send packet-out
stream_client.send(
    p4runtime_pb2.StreamMessageRequest(
        packet=p4runtime_pb2.PacketOut(payload=raw_bytes)
    )
)
```

<Note>
  If you do not pass `--cpu-port`, all `PacketIn` and `PacketOut` messages are silently ignored. You must set this flag to send or receive packets via the controller.
</Note>

## Idle timeout

P4Runtime supports idle timeout notifications on tables that declare the `support_timeout` table property in P4.

```p4
table my_table {
    key = { hdr.ipv4.dstAddr : exact; }
    actions = { forward; drop; }
    support_timeout = true;  // enable idle timeout
}
```

When a table entry has not been matched for a configured period, `simple_switch_grpc` sends an `IdleTimeoutNotification` message to the controller over `StreamChannel`. The controller can then decide whether to refresh or remove the entry.

<Tip>
  Idle timeout is useful for flow table implementations where entries should expire after inactivity, such as NAT or connection tracking tables.
</Tip>

## Connection and election

P4Runtime uses a primary/backup election model. Controllers establish a `StreamChannel` and send a `MasterArbitrationUpdate` message to claim primary role using an election ID.

```python
stream_client.send(
    p4runtime_pb2.StreamMessageRequest(
        arbitration=p4runtime_pb2.MasterArbitrationUpdate(
            device_id=0,
            election_id=p4runtime_pb2.Uint128(high=0, low=1),
        )
    )
)
```

Only the primary controller (the one with the highest election ID) can write state. Multiple controllers can read state simultaneously.
